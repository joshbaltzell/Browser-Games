# Phase 28: Kill Threshold Perks — Plan

## Overview

This phase adds five named kill-milestone perks that fire passively as `kills` accumulates during a run. The implementation touches one file (`game.js`) in three locations: a constant + global declaration in the globals section, a single reset line in `initGame()`, and two new code blocks inside `killEnemy()`. No new functions are required; all effects use existing globals (`maxPowerupsOnScreen`, `gemXpMult`) or `player` properties, and all notifications use the existing `spawnFloater()` call. The milestone check uses a `while` loop placed immediately after `kills++` so thresholds can never be silently skipped.

---

## Tasks

### T01 — Declare KILL_MILESTONES constant and milestoneIndex global

**File:** `game.js`
**Location:** Globals section, before `initGame()` (~line 323). Place alongside other global state declarations such as `surgeState`, `blackHoleActive`, `muted`.

**Code to add:**

```js
// --- Kill Threshold Perks (Phase 28) ---
const KILL_MILESTONES = [
  { kills:   50, name: 'VETERAN',   effect(p){ maxPowerupsOnScreen = (maxPowerupsOnScreen || 3) + 1; } },
  { kills:  150, name: 'SLAYER',    effect(p){ gemXpMult = (gemXpMult || 1) * 1.1; } },
  { kills:  350, name: 'DESTROYER', effect(p){ p.damage *= 1.05; } },
  { kills:  700, name: 'WARLORD',   effect(p){ p.dashCdBase = Math.max(0.4, (p.dashCdBase || 1.5) - 0.2); p.dashCd = p.dashCdBase; } },
  { kills: 1200, name: 'LEGEND',    effect(p){ p.bombProc = 0.05; } },
];
let milestoneIndex = 0;
```

**Notes:**
- `KILL_MILESTONES` is `const` because the array and its objects are never reassigned; only `milestoneIndex` changes.
- The WARLORD effect sets `player.dashCdBase` (so subsequent resets know the adjusted base) and then writes the same value to `player.dashCd`, which is the live cooldown property read by `executeDash()` (~line 485). `Math.max(0.4, ...)` prevents it going below 0.4.
- `maxPowerupsOnScreen` and `gemXpMult` are existing globals; the `|| 3` and `|| 1` guards make the effect safe regardless of initialization order.

### T02 — Reset milestoneIndex in initGame()

**File:** `game.js`
**Location:** Inside `initGame()` (~line 323), in the block that resets global counters and timers (near where `kills`, `elapsed`, `spawnTimer`, `combo`, `surgeState`, etc. are reset to their initial values).

**Code to add (one line):**

```js
milestoneIndex = 0;
```

**Notes:**
- Without this reset, `milestoneIndex` would carry over between runs and none of the milestones would fire on run 2+.
- Place it adjacent to the `kills = 0;` reset for logical grouping.

### T03 — Add milestone check and bomb proc in killEnemy()

**File:** `game.js`
**Location:** Inside `killEnemy(e)` (~line 1344), immediately after the line that increments `kills` (e.g., `kills++;` or `kills += 1;`). Insert both blocks before any existing post-kill logic (gem spawning, XP gain, particle effects) so the floater fires before other visual noise.

**Code to add:**

```js
// Kill threshold milestone check
while (milestoneIndex < KILL_MILESTONES.length &&
       kills >= KILL_MILESTONES[milestoneIndex].kills) {
  const m = KILL_MILESTONES[milestoneIndex];
  m.effect(player);
  spawnFloater(m.name, '#ffd700', 28);
  milestoneIndex++;
}

// LEGEND bomb proc
if (player.bombProc && Math.random() < player.bombProc) {
  spawnPowerup(e.x, e.y, 'bomb');
}
```

**Notes:**
- Using `while` (not `if`) handles the edge case where two kills fire in the same frame and `kills` skips past multiple thresholds simultaneously.
- `spawnFloater(text, color, size)` is an existing function used by the build-name HUD system; gold `'#ffd700'` at size 28 matches the visual weight of other significant mid-run events.
- `spawnPowerup(x, y, type)` is an existing function called elsewhere to place pickups; `'bomb'` is a valid key in `POWERUP_TYPES`.
- `player.bombProc` is `undefined` until the LEGEND milestone fires, so the falsy check makes this block a true no-op for all earlier milestones — no performance cost and no accidental procs.
- `e.x / e.y` uses the killed enemy's last position, which gives the bomb a natural, collectible spawn point.

## Verification

After implementing T01–T03, verify the following in a live browser run (open `index.html` via `file://`):

1. **VETERAN at 50 kills** — gold "VETERAN" floater appears; more powerups accumulate on screen than the prior default of 3.
2. **SLAYER at 150 kills** — gold "SLAYER" floater appears; XP gems visibly level the player faster.
3. **DESTROYER at 350 kills** — gold "DESTROYER" floater appears; `player.damage` logged to console just before and after the threshold shows a 5% increase.
4. **WARLORD at 700 kills** — gold "WARLORD" floater appears; dash recharges noticeably faster (0.2s shorter cooldown).
5. **LEGEND at 1200 kills** — gold "LEGEND" floater appears; bomb pickups begin spawning at enemy death positions at roughly 1-in-20 frequency.
6. **New run reset** — starting a second run triggers VETERAN again at 50 kills (milestoneIndex was reset in `initGame()`).
7. **No double-fire** — rapidly killing enemies near a threshold does not cause the same milestone to fire twice.
8. **Ordering** — milestones fire strictly in the order 50 → 150 → 350 → 700 → 1200; no earlier milestone fires after a later one.
9. **No JS errors** — browser console shows no errors during normal play, at each milestone, and when proc bombs are collected.
10. **Bomb procs are functional** — walking over a proc-spawned bomb triggers the normal `bomb` powerup explosion effect identical to a regular bomb pickup.
