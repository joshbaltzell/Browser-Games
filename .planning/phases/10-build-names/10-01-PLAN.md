# Phase 10: Build Names — Plan 10-01

## Goal

Detect recognized upgrade-combination patterns after each upgrade pick and flash a
large centered build-name floater for 2 seconds. Persist the current build name in
small HUD text. Pure presentation — zero gameplay changes. (FLAIR-01)

## Source-of-truth facts (verified against game.js, read before implementing)

- `UPGRADES` array (game.js lines 45–59) — the actual `id` values are:
  `damage`, `firerate`, `multishot`, `speed`, `maxhp`, `pickup`, `velocity`, `regen`,
  `splash`, `pierce`, `crit`, `orbital`, `lifesteal`.
  NOTE: the "+1 projectile" upgrade has id `multishot` and increments `player.projectileCount`.
  Build conditions that reference projectile count read `player.projectileCount` directly
  (NOT `upgradeCounts.multishot`), exactly as written in CONTEXT D-05/D-08/D-12.
- `player` object created in `initGame()` (game.js lines 110–139) — no `upgradeCounts`,
  `currentBuildName`, or `chainCount` exist yet.
- Phase 7 (Chain Lightning) is NOT implemented — there is no `player.chainCount`.
  Any build condition that touches chain MUST read it defensively (`player.chainCount || 0`).
- `chooseUpgrade(index)` (game.js lines 1104–1116) — `u.apply(player)` is on line 1107.
  This is the single integration point for both incrementing `upgradeCounts` and calling
  `checkBuildName()`.
- `spawnFloater(x, y, text, color, size)` (game.js lines 729–743) — `life`/`maxLife`
  are hard-coded to `0.8`. Needs an optional `lifeOverride` param (D-17).
- `updateFloaters()` (game.js lines 745–755) already animates by `life`/`maxLife`, so a
  longer-lived floater works with no other change.
- `updateHud()` (game.js lines 1064–1072) — HUD text update point.
- `dom` object (game.js lines 95–107) — add `buildName` reference.
- `index.html` HUD block (lines 13–28) — `.hud-stats` div (lines 24–27) is where the
  build-name element goes (just after it, inside `#hud`).
- `COLORS.gold` is `#fffb96` (game.js line 20). `W`/`H` are the canvas logical dimensions.
